<objective>
Backend infrastructure for inbound WhatsApp lot status queries. Store lot data server-side when lots arrive/dispatch, and expose a webhook endpoint for Twilio to forward incoming WhatsApp messages from parties.

Purpose: Fulfill WHATS-04 — party can text a WhatsApp message to check current lot quantity and receive an automated reply. The existing outbound notification flow (Phase 3) is extended to also persist lot data on the server, enabling server-side lot queries without requiring a separate lot CRUD API.

Output:
- `lots` table in SQLite with upsert/lookup helpers
- getPartyByPhone function for party lookup by phone number
- Enhanced notification route handlers that persist lot data on arrival/dispatch
- WhatsApp webhook handler (POST /api/whatsapp/webhook) that accepts Twilio inbound messages, looks up the party, queries their lots, and replies via TwiML
- Webhook route wired into Express before the authenticated notification routes
</objective>

  <files>server/src/db.js</files>
  <action>
    Add a `lots` table to the schema in `getDb()`:

    ```sql
    CREATE TABLE IF NOT EXISTS lots (
      id TEXT PRIMARY KEY,
      lot_number TEXT UNIQUE NOT NULL,
      party_name TEXT NOT NULL,
      quantity REAL NOT NULL,
      fabric_type TEXT DEFAULT '',
      status TEXT NOT NULL DEFAULT 'active',
      created_at TEXT DEFAULT (datetime('now')),
      dispatched_at TEXT
    )
    ```

    Add these exported helper functions at the bottom of the file (after the existing upsertParty):

    1. `upsertLot(lotNumber, partyName, quantity, fabricType)` — INSERT or UPDATE lot by lot_number.
       Use: `INSERT INTO lots (id, lot_number, party_name, quantity, fabric_type) VALUES (?, ?, ?, ?, ?) ON CONFLICT(lot_number) DO UPDATE SET quantity = excluded.quantity, party_name = excluded.party_name, fabric_type = excluded.fabric_type`
       Generate id with `crypto.randomUUID()`.
       This ensures the same lot_number cannot be duplicated — if a lot is re-saved (e.g., correction), it updates in place.

    2. `markLotDispatched(lotNumber)` — UPDATE lots SET status = 'dispatched', dispatched_at = datetime('now') WHERE lot_number = ?.

    3. `getPartyByPhone(phone)` — SELECT * FROM parties WHERE phone = ?.
       Returns the party row or null.
       This is needed by the webhook handler to identify which party sent the WhatsApp message.

    4. `getTotalLotQuantityByParty(partyName)` — SELECT COALESCE(SUM(quantity), 0) FROM lots WHERE party_name = ? AND status = 'active'.
       Returns a number (total kg of active lots for the party).

    5. `getActiveLotCountByParty(partyName)` — SELECT COUNT(*) FROM lots WHERE party_name = ? AND status = 'active'.
       Returns a number.

    All helpers use `getDb()` to get the database connection. Follow the exact same import/export pattern as existing helpers (no default export for functions, named exports only).

    Do NOT add a seed for the lots table — lots are populated dynamically when the frontend creates them.

    2. composeDispatchMessage(...)

    These modifications are synchronous (better-sqlite3 is synchronous) and non-blocking. They happen before the async sendWhatsApp call. If the upsert/update fails, the error propagates via express-async-errors (express-async-errors is already imported in index.js, and both handlers are async, so errors are caught automatically). However, since these are side-effects of the notification flow, wrap with try/catch as a safety measure and log the error without blocking the WhatsApp send:

    ```javascript
    try {
      upsertLot(lotNumber, party.name, quantity, fabricType || '');
    } catch (err) {
      console.error(`[Lot Sync] Failed to store lot ${lotNumber}:`, err.message);
    }
    ```

    And similarly for markLotDispatched:

    ```javascript
    try {
      markLotDispatched(lotNumber);
    } catch (err) {
      console.error(`[Lot Sync] Failed to mark lot ${lotNumber} as dispatched:`, err.message);
    }
    ```

    Rationale: The notification flow (WhatsApp send) is the primary concern. Lot data persistence is a supporting side-effect. Failing to persist should not prevent the notification from being sent.

    Export a single async function `handleWhatsAppWebhook`:

    ```javascript
    export async function handleWhatsAppWebhook(req, res) {
      const { From, Body } = req.body;

      if (!From) {
        return res.status(400).send('Missing From parameter');
      }

      // Strip "whatsapp:" prefix from the From field (e.g., "whatsapp:+919000000001")
      const phone = From.replace('whatsapp:', '');

      // Log incoming message in all modes
      console.log(`[WhatsApp Webhook] Incoming from ${phone}: "${Body}"`);

      // Look up party by phone number
      const party = getPartyByPhone(phone);

      if (!party) {
        console.log(`[WhatsApp Webhook] No party found for phone ${phone}`);
        const replyText = 'Thank you for contacting TextileFlow. Your WhatsApp number is not registered in our system. Please contact the factory to register your number for lot updates.';

        if (process.env.WHATSAPP_MODE === 'twilio') {
          const twilio = (await import('twilio')).default;
          const MessagingResponse = twilio.twiml.MessagingResponse;
          const twiml = new MessagingResponse();
          twiml.message(replyText);
          res.type('text/xml');
          return res.send(twiml.toString());
        }

        // Dev mode: log the intended reply
        console.log('[WhatsApp Webhook Dev] Reply:', replyText);
        return res.json({ received: true, mode: 'log', reply: replyText });
      }

      // Party found — query their lot data
      const totalQuantity = getTotalLotQuantityByParty(party.name);
      const activeLotCount = getActiveLotCountByParty(party.name);

      const replyText = [
        `Dear ${party.name},`,
        '',
        `Your current lot status at TextileFlow:`,
        `  • Active lots: ${activeLotCount}`,
        `  • Total quantity in factory: ${totalQuantity} kg`,
        '',
        `Thank you.`,
        `— TextileFlow MES`,
      ].join('\n');

      console.log(`[WhatsApp Webhook] Party: ${party.name}, Active lots: ${activeLotCount}, Total: ${totalQuantity}kg`);

      if (process.env.WHATSAPP_MODE === 'twilio') {
        const twilio = (await import('twilio')).default;
        const MessagingResponse = twilio.twiml.MessagingResponse;
        const twiml = new MessagingResponse();
        twiml.message(replyText);
        res.type('text/xml');
        return res.send(twiml.toString());
      }

      // Dev mode: log the intended reply
      console.log('[WhatsApp Webhook Dev] Reply:', replyText);
      console.log('[WhatsApp Webhook Dev] ---');
      res.json({ received: true, mode: 'log', reply: replyText });
    }
    ```

    Key design decisions:
    - Uses TwiML `<Response><Message>` for Twilio production mode (Twilio sends the reply directly, no extra REST API call needed).
    - In dev/log mode, returns JSON so the endpoint can be tested with curl without Twilio.
    - The dynamic `import('twilio')` for TwiML keeps the module lightweight when twilio is not used (lazy import pattern matching whatsapp.js).
    - Any message from a registered number triggers a lot status reply (simple v1 approach — no intent parsing needed).
    - Unknown numbers get a generic "not registered" response.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Twilio -> Webhook | Unauthenticated HTTP POST from Twilio containing user message (From phone, Body text) |
| Server -> Twilio | Outbound HTTPS response via TwiML containing lot status data (party name, quantity) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | POST /api/whatsapp/webhook | accept | Twilio webhook signature validation is deferred to a future enhancement. Risk is low: unauthenticated callers can at worst cause a reply to a registered party with stale lot data. No data exfiltration possible — the reply only contains data the party already owns. |
| T-04-02 | Information Disclosure | Lot status reply | mitigate | Only registered phone numbers (in parties table) receive lot data. Unregistered numbers get a generic "not registered" response with zero production data. |
| T-04-03 | Tampering | Webhook Body field | accept | The Body field is not used for anything other than logging. All reply data comes from the server database (lots table), not from the incoming message. Body manipulation has no effect on the response. |
| T-04-04 | Denial of Service | Webhook endpoint | accept | Dev mode (default) has no Twilio costs. Production deployments can add express-rate-limit later if needed. Factory floor context makes abuse unlikely. |
</threat_model>
