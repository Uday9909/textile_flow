  <action>
    Create three modules for the auth system. Per FEATURES.md T1-T6, the auth API supports login, logout, token refresh, and current-user lookup.

    **server/src/auth.js** — JWT utility module using jose:
    - `createAccessToken(user)` — creates JWT signed with JWT_ACCESS_SECRET using HS256, expires in JWT_ACCESS_EXPIRES_IN seconds. Payload: `{ sub: user.id, email: user.email, name: user.name, role: user.role, department: user.department }`. Pin algorithm in sign: `{ alg: 'HS256' }`.
    - `verifyAccessToken(token)` — verifies JWT using JWT_ACCESS_SECRET. Pin algorithm: `{ algorithms: ['HS256'] }`. Returns decoded payload or throws.
    - `createRefreshToken(userId)` — creates JWT signed with JWT_REFRESH_SECRET using HS256, expires in JWT_REFRESH_EXPIRES_IN seconds. Payload: `{ sub: userId, type: 'refresh' }`. Pin algorithm.
    - `verifyRefreshToken(token)` — verifies using JWT_REFRESH_SECRET. Pin algorithm. Returns decoded payload or throws.
    - Per PITFALLS.md rule #3: ALWAYS pin algorithm in verify calls. This is non-negotiable.
    - Per PITFALLS.md rule #2: access tokens expire in 15 minutes, refresh tokens in 7 days.

    **server/src/middleware/authenticate.js** — Auth middleware:
    - Extract token from `Authorization: Bearer <token>` header
    - Call verifyAccessToken(token) 
    - On success: attach `req.user = { id, email, name, role, department }` (from decoded token)
    - On failure: return 401 `{ error: 'Authentication required' }` (generic message, no detail)
    - Export as named export `authenticate`
    - Per PITFALLS.md rule #9: generic error messages on all auth failures

    **server/src/routes/auth.js** — Express Router with 4 endpoints:

    **POST /api/auth/login** — Rate-limited (5 attempts per 15 min per IP per PITFALLS.md rule #8):
    1. Validate body has `email` and `password` (string, non-empty)
    2. Call `getUserByEmail(email)` from db module
    3. Call `verifyPassword(password, user.password_hash)` 
    4. If user not found OR password invalid: return 401 `{ error: 'Invalid email or password' }` (generic, per pitfall #9)
    5. On success: create access token (15 min) and refresh token (7 days)
    6. Store refresh token hash in DB via storeRefreshToken
    7. Set refresh token as httpOnly cookie: name `refreshToken`, value is the JWT refresh token, httpOnly: true, secure: process.env.NODE_ENV === 'production', sameSite: 'strict', path: '/api/auth', maxAge: 604800 (7 days)
    8. Return 200 JSON: `{ user: { id, email, name, role, department }, accessToken, accessTokenExpiresIn: 900 }`
    9. Do NOT return the refresh token in the response body — it is cookie-only

    **POST /api/auth/refresh** — No rate limit:
    1. Read `refreshToken` from cookies (req.cookies.refreshToken)
    2. If missing: return 401 `{ error: 'Session expired' }`
    3. Verify the refresh token with verifyRefreshToken
    4. Look up the stored refresh token hash in DB — if not found, return 401
    5. Delete the used refresh token from DB (rotation)
    6. Call getUserById(decoded.sub) — if user not found, return 401
    7. Create new access token and new refresh token
    8. Store new refresh token hash in DB
    9. Set new refresh cookie (same cookie config as login)
    10. Return 200 with new access token: `{ accessToken, accessTokenExpiresIn: 900 }`

    **POST /api/auth/logout**:
    1. Read `refreshToken` from cookies
    2. If present: verify it, delete from DB, clear the cookie (set same name/path with maxAge: 0)
    3. Return 200 `{ message: 'Logged out' }`
    4. This is idempotent — succeeds even if no cookie present

    **GET /api/auth/me** — Protected (uses authenticate middleware):
    1. Returns 200 JSON: `{ user: { id, email, name, role, department } }` from req.user
    2. This lets the frontend verify the current session on page load

    What to avoid and WHY:
    - Do NOT use `jsonwebtoken` — jose is mandated per STACK.md. jose uses async verify/sign.
    - Do NOT return the refresh token in response body — it is httpOnly cookie only, per PITFALLS.md rule #1
    - Do NOT expose error details on login failure — generic "Invalid email or password" per PITFALLS.md rule #9
    - Do NOT accept token `alg` from the token header — pin algorithm in verify per PITFALLS.md rule #3
    - Do NOT store raw refresh tokens — store SHA-256 hash in DB for comparison
    - Do NOT use `res.cookie` with `secure: true` in dev (no HTTPS) — use secure only in production
  </action>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | POST /api/auth/login | mitigate | bcrypt password verification (10 rounds); generic error messages prevent email enumeration |
| T-01-02 | Tampering | JWT tokens | mitigate | jose with pinned HS256 algorithm; verify rejects tokens with different alg header |
| T-01-03 | Information Disclosure | Error responses | mitigate | All auth errors return generic messages; no stack traces in production |
| T-01-04 | Elevation of Privilege | GET /api/auth/me | mitigate | authenticate middleware verifies token and attaches req.user; no path traversal |
| T-01-05 | Repudiation | Refresh token rotation | mitigate | refresh tokens are one-time-use; old tokens invalidated on rotation |
| T-01-06 | Denial of Service | Login endpoint | mitigate | express-rate-limit: 5 attempts per 15 min per IP on /api/auth/login |
| T-01-07 | Tampering | Refresh token cookie | mitigate | httpOnly + SameSite=Strict prevent XSS/CSRF token theft; secure flag in production |
| T-01-SC | Tampering | npm install (better-sqlite3, jose, etc.) | mitigate | npm audit on install; human verify at first run |
</threat_model>
