  <action>
    Create two modules: AuthContext (auth state + actions) and api.js (authenticated fetch wrapper).

    **src/api.js** — Fetch wrapper module (no React dependency, plain module):
    - Export setAccessToken(token) and getAccessToken() — store access token in a module-level variable (NOT localStorage — per PITFALLS.md rule #1)
    - Export clearAccessToken() to clear the module-level token
    - Export a refreshAccessToken() function that calls POST /api/auth/refresh with credentials: 'include' (sends cookies), reads accessToken from response, stores it via setAccessToken, returns the new token
    - Export a generic api(url, options) function that:
      1. Calls fetch(url, { ...options, credentials: 'include', headers: { ...defaultHeaders, Authorization: getAccessToken() ? Bearer plus token : undefined, ...options?.headers } })
      2. If response is 401: try refreshAccessToken(), then retry the original request once with the new token
      3. If refresh also fails (refresh token expired/invalid): clearAccessToken(), return the 401 response
      4. Otherwise return the response
    - Export specific convenience wrappers: api.login(email, password) — POST /api/auth/login with JSON body, returns parsed response; api.logout() — POST /api/auth/logout, returns parsed response; api.me() — GET /api/auth/me, returns parsed response; api.refresh() — POST /api/auth/refresh, returns parsed response
    - Do NOT use axios — native fetch only per STACK.md
    - Base URL default: http://localhost:3001. Use import.meta.env.VITE_API_URL if defined, else http://localhost:3001.
    - Do NOT set credentials: 'include' on login — the server sets the refresh cookie in the login response; subsequent calls include it automatically

    **src/context/AuthContext.jsx** — React Context + useReducer pattern (matching existing AppContext style):

    State shape has three fields: user (null or object with id, email, name, role, department), status (one of 'loading', 'authenticated', or 'unauthenticated'), and error (null or string).

    Action types:
    - AUTH_LOGIN_START — set status to loading, clear error
    - AUTH_LOGIN_SUCCESS — set user object, status to authenticated, clear error
    - AUTH_LOGIN_FAILURE — set error string, status to unauthenticated
    - AUTH_LOGOUT — set user to null, status to unauthenticated, clear error
    - AUTH_RESTORE_SESSION — set user object, status to authenticated (for page load recovery)

    Initial state: user null, status 'loading', error null.

    Exports:
    - AuthProvider component — wraps children
    - useAuth() hook — returns object with user, status, error, login function, logout function

    AuthProvider implementation:
    1. On mount (useEffect): try to restore session by calling api.me() through the 401-interceptor mechanism built into api.js. If it succeeds: dispatch AUTH_RESTORE_SESSION with the user. If it fails (401): dispatch AUTH_LOGOUT (means no valid session). The api.js 401 interceptor will trigger silent refresh on the first call.
    2. Provide login function: takes email and password, calls api.login(), on success stores accessToken via setAccessToken and dispatches AUTH_LOGIN_SUCCESS with user, on failure dispatches AUTH_LOGIN_FAILURE with error message.
    3. Provide logout function: calls api.logout(), clears access token via clearAccessToken(), dispatches AUTH_LOGOUT.
    4. While status is 'loading' (initial auth check on page load), render a minimal centered loading state (small "Loading..." text or spinner div). Per ARCHITECTURE.md: loading state is needed for the initial auth check on page load.

    What to avoid and WHY:
    - Do NOT store access token in localStorage — per PITFALLS.md rule #1, localStorage JWT is the #1 auth pitfall. Use module variable (in memory) only.
    - Do NOT modify AppContext — AuthContext is a separate context for different lifecycle and security concerns per STACK.md key decisions
    - Do NOT import any UI components into AuthContext — it is a pure data/state layer
    - Do NOT use axios — native fetch is sufficient per STACK.md
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser memory -> AuthContext | Module-level access token is only in memory; cleared on refresh (intentional design — forces refresh cookie check) |
| browser -> Express API | AuthContext and api.js communicate with Express backend over HTTP |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-08 | Information Disclosure | api.js module variable | mitigate | Access token stored in module closure, not localStorage; cleared on page close and refresh |
| T-01-09 | Spoofing | LoginPage form input | mitigate | Email input validated as type="email" before sending to API; server validates server-side |
| T-01-10 | Elevation of Privilege | ProtectedRoute redirect bypass | mitigate | Backend enforces auth independently (authenticate middleware from Plan 01); frontend redirect is UX-only, not security boundary |
| T-01-11 | Repudiation | Logout flow | mitigate | Logout clears both in-memory token (clearAccessToken) and server-side refresh token (api.logout); idempotent |
</threat_model>
